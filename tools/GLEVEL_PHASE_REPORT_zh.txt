  val_acc：0.5397→0.476（约 −6.4pt）；Medium recall：0.176→0.41（+0.23）；High recall：0.840→0.48（−0.36）。

四、问题机制说明（为何「加东西」反而 acc 掉）
----------------------------------------------------------------
1) 验证集极小（n=63）
   早停与 select_best 对个别样本敏感；best_epoch 在不同 run 间波动大是预期现象。

2) 指标冲突
   select_best=balanced_acc（A）与 macro_f1 + manual 权重（C）优化的目标不同；manual 权重与关闭平衡采样会改变
   梯度分布，往往压低「多数易分类（此处多为 High）」的置信度，从而拉低 acc 但抬高 Medium。

3) 时序模块容量与数据量
   450 条样本、双向 GRU + 注意力 + 跨模态 Transformer 参数量偏大，易过拟合或优化陷入较差区域；B 相对 A 的整体
   下降符合「小数据上复杂归纳偏置未必有益」的经验。

4) val_ce 不可跨配置直接比
   label_smoothing、class_weight 均改变 CE 标度；C 在 best 点 val_ce≈1.88 高于 A 的约 1.13 不代表「更差」，
   仅说明损失数值不可横向对比。

5) 测试链路
   若 FEAT_TEST 或测试 id 的 Nanbeige 文本缺失，则无法生成提交文件；需在服务器上先跑 tools/preflight_test_submission.sh
   或 route_a 第四节。

五、推荐下一步（按优先级）
----------------------------------------------------------------
P0  打通测试特征：确认 FEAT_TEST 与 TEXT_TEST_DIR 对 130 个测试 id 的 6 题×模态覆盖，否则无法真实迭代提交策略。

P1  缓解 n=63 方差：运行 tools/run_kfold_multimodal_plus.sh 或 run_multi_seed_multimodal.sh，用均值±方差选配置，
    避免被单次 val 误导。

P2  在 A 与 C 之间插值：使用 MM_MEDIUM_BOOST=1（sampler_medium_boost 默认 1.5）保留 A 的 select_best 与平衡采样，
    观察 Medium recall 是否上升而 acc 下降少于 C。

P3  时序消融：仅在 K 折上对比 MM_TEMPORAL 开/关；若稳定为负，暂时关闭时序，把算力留给特征与集成。

P4  赛方指标对齐：若最终评测为 accuracy，select_best 优先 val_ce 或 balanced_acc，并谨慎叠加 medium 权重。

P5  上游特征：检查转写质量、题级对齐与 hand/音视特征版本；可分性上限往往由特征决定。

----------------------------------------------------------------
已用改进（相对最初基线）：赛方 train/val 多模态 SharedMLPwEnsemble；cross_modal_attn；dropout / label_smoothing /
  weight_decay；WeightedRandomSampler；Plateau + 长早停与 early_stop_min_epochs；Nanbeige 文本维（与服务器 export 一致）；

  下降；**强 Medium 聚焦（C）** 拉高 Medium recall 但 acc 与 macro_F1 低于 A。

当前主要问题：
  (1) **偏类**：A 下 High recall 84%、Medium 仅 17.6%，错分集中在 Med→High、Low→High。
  (2) **验证规模小**：63 条使单次 run 方差大，需 K 折/多种子再下结论。

  (4) **目标冲突**：抬 Medium 的配置会伤 acc，需按赛方指标选折中（如 MM_MEDIUM_BOOST）。

突破重点：P0 补全测试特征；P1 K 折/多种子；P2 A 与 C 之间试 MM_MEDIUM_BOOST；P3 时序仅在有统计增益时保留；
  P4 上游特征与转写质量。
